# 开放世界交互规则

## 核心原则

1. **不丢失上下文**：用户插入新需求时，保留当前流程状态，处理完插入后可恢复
2. **不假设意图**：对模糊插入主动确认，不自行脑补
3. **不阻塞主流程**：插入处理应尽量简短，避免让用户等待过久
4. **不重复确认**：已确认的信息不再次询问

## 插入类型与处理策略

### 时间变更类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 今天没空 | "我今天没空了" | 查询用户日程空档 + 门店档期，推荐最近可用时间，确认后改约并更新宠物专车 |
| 换个时间 | "换个时间吧" | 询问偏好时间段，查询档期，推荐方案 |
| 晚点送回 | "晚点送回来" | 确认新送回时间，call_service(pet_transport_mock, modify_transport, ...) |

### 服务变更类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 加服务 | "加个刷牙吧" | 查询差价和额外耗时，检查档期是否允许延长，确认后修改预约 |
| 减服务 | "不修毛了" | 计算差价退还，确认后修改预约 |
| 换服务 | "换个套餐" | 展示可选套餐，确认后修改预约 |

### 商品追加类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 顺便买 | "顺便买袋狗粮" | 查询商品、库存和配送，确认后创建新订单 |
| 独立买 | "我家猫粮快没了" | 启动商品购买子流程 |
| 换商品 | "换个牌子的" | 展示替代商品，确认后修改订单 |

### 取消类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 取消预约 | "别洗了" | 确认取消意图，取消预约和宠物专车 |
| 取消订单 | "不要了" | 确认取消意图，取消订单和配送 |
| 取消打车 | "我自己送过去" | call_service(pet_transport_mock, cancel_transport, ...)，保留预约 |

### 门店变更类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 换近的 | "换近一点的店" | 重新查询门店，比较距离、档期、库存 |
| 换便宜的 | "有没有更便宜的" | 查询活动价格，推荐性价比方案 |

### 异常情况类

| 用户表达 | 示例 | 处理策略 |
| --- | --- | --- | 
| 宠物状态 | "猫今天应激了" | 暂停服务，建议改期或人工确认 |
| 紧急情况 | "狗吐了" | 建议立即就医，暂停当前预约流程 |
| 天气影响 | "下大雨了" | 提醒宠物专车可能延迟，建议改期 |

## 插入处理流程

```
1. 识别插入类型
2. 判断是否影响当前流程
3. 如果影响：
   a. 暂存当前流程状态
   b. 处理插入需求
   c. 根据结果更新或恢复当前流程
4. 如果不影响：
   a. 快速处理插入需求
   b. 继续当前流程
```

## 多插入处理

当用户连续插入多个需求时：

1. 按优先级排序：取消 > 异常 > 时间变更 > 服务变更 > 商品追加
2. 逐个处理，每处理完一个更新流程状态
3. 全部处理完后，给出综合确认摘要
